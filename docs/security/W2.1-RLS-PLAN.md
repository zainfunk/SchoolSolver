# W2.1 — Re-architect club-scoped RLS

**Status:** plan, awaiting human review (per W2.1 directive: "Wait for human review of the plan before writing the migration.")

**Closes:** finding C-4 in `ClubIt-Security-Assessment.md`.

**Migration target:** `supabase/migrations/0002_club_membership_rls.sql` + `.down.sql`.

---

## 1. The problem

Every table in the club domain currently scopes `SELECT` to `app.club_in_scope(target_club_id)`, which returns true if the club's `school_id` matches the caller's `school_id`. **That is school-wide, not membership-scoped.** A student in school X can read every club's chat messages, attendance, polls, and forms — even clubs they're not in.

The assessment named ~14 affected tables. Confirmed by grep against `supabase/schema.sql`: **16** tables call `club_in_scope`, `poll_in_scope`, or `form_in_scope`. Listed below.

## 2. Helper functions to add (migration step 1)

The current schema already has:

- `app.current_user_id()` — `auth.jwt()->>'sub'`
- `app.current_school_id()` — `users.school_id` for `current_user_id`
- `app.is_superadmin()` / `app.is_school_admin()`
- `app.club_member(target_club_id, target_user_id default current)` — already exists
- `app.club_manager(target_club_id)` — `is_school_admin OR clubs.advisor_id = current`
- `app.club_in_scope(target_club_id)` — **the school-wide one we're getting away from**

Adding two new helpers:

```sql
-- True if `target_user_id` holds any leadership_positions row for the club.
create or replace function app.club_leader(
  target_club_id text,
  target_user_id text default app.current_user_id()
) returns boolean
language sql stable security definer
set search_path = public, pg_temp
as $$
  select exists (
    select 1 from leadership_positions lp
    join clubs c on c.id = lp.club_id
    where lp.club_id = target_club_id
      and lp.user_id = target_user_id
      and c.school_id = app.current_school_id()
  );
$$;

-- True if `target_user_id = clubs.advisor_id`. Cheap shortcut so policies
-- can express "advisor only" without going through club_manager (which
-- also matches school admins).
create or replace function app.club_advisor(
  target_club_id text,
  target_user_id text default app.current_user_id()
) returns boolean
language sql stable security definer
set search_path = public, pg_temp
as $$
  select exists (
    select 1 from clubs c
    where c.id = target_club_id
      and c.advisor_id = target_user_id
      and c.school_id = app.current_school_id()
  );
$$;
```

Both `SECURITY DEFINER` so they run with the function-owner's privilege (planned: `service_role`, schema-qualified, search_path locked). This is needed because RLS policies invoking these functions must read tables (`memberships`, `clubs`, `leadership_positions`) that themselves have RLS — without `SECURITY DEFINER` we get infinite recursion or visibility denial.

The existing `club_in_scope` function will be **kept**. A handful of tables legitimately use school-wide scope (clubs, leadership_positions, club_social_links, meeting_times — see §4 below). It just stops being the default.

## 3. Actor classes

Used to label the cells of the matrix:

| Code | Actor | How RLS recognizes |
|---|---|---|
| **SUPER** | superadmin | `app.is_superadmin()` |
| **ADMIN** | school admin (same school as target) | `app.is_school_admin()` AND target's school_id matches caller's |
| **ADV** | club advisor (`clubs.advisor_id = caller`) | `app.club_advisor(club_id)` |
| **LEAD** | leadership_positions row for caller | `app.club_leader(club_id)` |
| **MEM** | regular member (memberships row) | `app.club_member(club_id)` (already includes ADV & ADMIN) |
| **USER** | authenticated, same school, not in club | `users.school_id = caller.school_id` only |
| **OTHER** | different school | the implicit deny |
| **SELF** | row's `user_id`/`sender_id`/`voter_id` = caller | direct equality |

`MEM` is a superset of `LEAD` and `ADV` in the existing helper (`club_manager OR exists membership`); for clarity in the matrix below, "MEM" means *any* membership-or-stronger; "USER" means strictly the in-school-non-member case.

### 4.2 Tables I'm leaving school-wide (with justification)

| # | Table | Why school-wide stays |
|---|---|---|
| 13 | `clubs` | Directory of clubs is the central UX. Strictly school-scoped, no membership requirement. |
| 14 | `leadership_positions` | "Who is president of robotics?" is part of a club's public profile; lets students decide whether to join. |
| 15 | `club_social_links` | Public marketing links by definition. |
| 16 | `meeting_times` | Schools advertise meeting times to attract members. |

These keep the existing `app.club_in_scope` SELECT. Their write policies (`manage`) already require `club_manager`. **Unchanged in this migration.**

### 4.3 Tables out of W2.1 scope

`user_profiles`, `user_overrides`, `user_privacy_settings`, `user_badges` — user-keyed; covered by `app.user_in_scope`. Tightened in a later wave if needed.
`school_elections`, `election_candidates`, `election_votes` — school-scoped, not club-scoped. `election_votes` SELECT is addressed in W2.2 (secret ballot).
`schools`, `users`, `clubs`, `admin_settings`, `issue_reports`, `notifications`, `subscriptions`, `payment_events` — already tenant-scoped or server-only writes.

## 5. Migration plan (file order)

1. **`supabase/migrations/0002_club_membership_rls.sql`** — single transaction:
   1. Create `app.club_leader()` and `app.club_advisor()`.
   2. For each of the 12 tables in §4.1: `drop policy if exists` + `create policy` with the new clauses. Replace `app.club_in_scope` with the right combination of `app.club_member`, `app.club_manager`, `app.is_superadmin`, etc.
   3. `commit`.
2. **`supabase/migrations/0002_club_membership_rls.down.sql`** — restores the policies from `schema.sql` lines 746-1027.

Both files idempotent (`if exists` / `if not exists`).

## 6. UX side effects you need to know about

These are real and need acknowledgment before SQL is written.

1. **Membership counts on the explore-clubs page** (`app/clubs/page.tsx`). Today's UI shows "12 members" next to each club. With the strict policy in §4.1 row 10, a non-member querying `memberships` for that club gets zero rows. Two options:
   - (a) Add a server endpoint `GET /api/school/clubs/counts` that returns `{[clubId]: count}` using the service-role client. Cheap, doesn't expose identities. **My recommendation.**
   - (b) Denormalize a `member_count` column on `clubs` updated by a trigger.
   - Will land alongside the migration in the same wave, in a separate commit.

2. **Realtime subscriptions** (`lib/realtime.ts`). The realtime channel for `chat_messages` currently delivers every message in the school to every connected client (RLS gates it but RLS is school-wide). After the migration, the same RLS gates it correctly — clients no longer receive messages they couldn't fetch. **No client code change needed**, but I'll write a regression test that confirms a non-member receives zero realtime events from a club they're not in.

3. **"What's happening at school" feed** (if any). The dashboard route shows pinned news from clubs the user is a member of (already filtered correctly server-side). After migration, a non-member querying `club_news` directly gets nothing, but the dashboard loads from the server with service-role client, so the dashboard UI is unchanged. Confirmed by re-reading `app/api/school/dashboard/route.ts`.

4. **Forms & polls discoverability.** Today a school-wide visible poll is theoretically discoverable; after the change it's invisible to non-members. This is the *intended* fix. If product wants discoverable cross-club polls, that's a feature request, not a security regression.

## 7. Test approach

`tests/security/rls/test_w2_1_club_rls.spec.ts` — Vitest, needs the same `SUPABASE_TEST_*` env vars as W1.4.

Seed fixture (one-time, in `beforeAll`):

- Two schools: `X`, `Y`.
- In school X: two clubs, `CX1` and `CX2`. In school Y: one club, `CY1`.
- Users (Clerk-style text IDs):
  - `superA` (role=superadmin, school_id=null)
  - `adminX` (admin, school X) / `adminY` (admin, school Y)
  - `advisorX1` (advisor, school X) — `clubs.advisor_id` for `CX1`
  - `leaderX1` (student, school X) — leadership_position in `CX1`
  - `memberX1` (student, school X) — membership in `CX1`
  - `userX_no` (student, school X) — no membership
  - `userY` (student, school Y) — no relation to school X

That's 8 actors. Apply to each of the 12 affected tables × 2-4 verbs each. Roughly **80 test cases**, written as a parametrized loop driven by a JSON-shaped table that mirrors §4.1.

The test must run against a database with both `0001_users_rls_lockdown.sql` and `0002_club_membership_rls.sql` applied. The RLS README will document this.
