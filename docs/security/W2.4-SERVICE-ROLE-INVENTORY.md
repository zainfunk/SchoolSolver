# W2.4 — Service-role usage inventory

Closes finding C-8 (architecture: "RLS is decorative because every route uses
the service-role client and re-implements authorization in TypeScript").

This file is the audit trail for **every** server-side use of
`createServiceClient()` (which uses `SUPABASE_SERVICE_ROLE_KEY` and bypasses
RLS). Two purposes:

1. **Justification.** Each row says why the route bypasses RLS. New
   service-role usage requires adding a row here AND a code-review tag in
   the PR.
2. **Refactor list.** Rows marked `Category: B` are routes that should
   instead use the authenticated client (`createAuthedServerClient`) and
   let RLS authorize. These are scheduled for refactor in this commit and
   follow-ups.

## Categories

- **A — Justified service-role usage.** The route either has no caller
  identity (webhooks, public token redemption), is bootstrapping the
  caller's identity (first-login sync, onboarding), needs cross-tenant
  reach (superadmin tools), or writes a column the W1.4 trigger blocks
  for non-service-role (role, school_id changes — done via dedicated admin
  endpoints, not the browser).
- **B — Should use authenticated client.** RLS already covers the access
  control; bypassing it adds nothing but masks bugs.
- **C — Mixed.** Large multi-action handlers where some actions need
  service role and some don't. Refactor to split or to use service role
  only on the privileged actions.

## How to add a new service-role route

If you write a new route that uses `createServiceClient()`:

1. Add a row to the table above with category, justification, and reviewer.
2. In the PR description, reference this file by line number.
3. Code-review checklist (`docs/security/CODE_REVIEW_CHECKLIST.md`, scaffolded
   by W4 in a follow-up): "service_role usage justified per
   W2.4-SERVICE-ROLE-INVENTORY.md row [N]?"

If you can use the authenticated client instead — do. RLS is the only
authorization layer that doesn't drift.
